# 项目展示说明

## 项目定位

本项目是一个面向物流供应链场景的 RAG + Agent 工作流应用。系统使用 Dify 负责知识库检索和 Chatflow 流程编排，使用 FastAPI 提供订单工具接口，实现物流业务知识问答、订单状态查询、异常处理建议生成，并通过测试集评估回答效果。

项目重点展示大模型应用从业务知识整理、RAG 检索、工具调用、Prompt 约束到效果评测的完整链路。

## 业务问题

物流供应链场景中同时存在静态规则和动态数据：

- 静态规则包括仓储入库、出库、冷链运输、退换货、投诉处理、库存盘点等 SOP。
- 动态数据包括订单状态、当前物流节点、预计送达时间、异常类型等。
- 异常处理需要同时参考订单事实和业务 SOP，单独依赖知识库或单独调用接口都不完整。

因此，项目将业务 SOP 放入 Dify Knowledge，将订单状态放在 FastAPI 工具接口中，再通过 Chatflow 根据用户意图选择处理路径。

## 系统架构

项目分为三层：

- 知识层：`docs/` 中的物流供应链 Markdown 文档，作为 Dify Knowledge 上传源。
- 服务层：`order_api/` 中的 FastAPI 订单接口，提供订单查询和异常分析能力。
- 编排层：Dify Chatflow，负责任务分类、参数提取、知识检索、HTTP 工具调用和最终回答生成。

整体流程如下：

```text
用户问题
  -> Question Classifier
  -> 知识库问答 / 订单状态查询 / 异常处理建议 / 其他问题
  -> Knowledge Retrieval 或 HTTP Request
  -> LLM 结构化整理
  -> Answer
```

## 核心分支

### 知识库问答

适用于仓储、运输、冷链、退换货、投诉、盘点等规则类问题。

```text
用户问题 -> 问题分类器 -> Knowledge Retrieval -> LLM_知识问答 -> Answer
```

该分支要求模型仅依据知识库上下文回答，并在信息不足时明确说明未找到依据。

### 订单状态查询

适用于包含订单号的状态查询问题。

```text
用户问题 -> 问题分类器 -> 参数提取器 -> HTTP Request /orders/query -> LLM_整理订单状态 -> Answer
```

该分支通过 `Parameter Extractor` 提取 `order_id`，再调用 FastAPI 获取订单状态、当前节点、预计送达时间和异常类型。

### 异常处理建议

适用于订单延迟、温度异常、货物破损、客户投诉等问题。

```text
用户问题 -> 问题分类器 -> 参数提取器 -> HTTP Request /orders/analyze
  -> Knowledge Retrieval_异常SOP -> LLM_生成异常处理建议 -> Answer
```

该分支同时使用订单接口结果和知识库 SOP，输出订单概况、异常判断、建议处理步骤、客户沟通话术和是否需要人工升级。

## RAG 配置

知识库问答与异常处理分支采用 Dify Knowledge 检索，推荐配置如下：

| 分支 | Retrieval Method | Top K | Score Threshold |
|---|---|---:|---:|
| 知识库问答 | Hybrid Search | 5 | 0.35 |
| 异常处理建议 | Hybrid Search | 5 | 0.35 |

异常处理建议分支可启用 `Qwen/Qwen3-Reranker-8B` 进行重排，提高 SOP 片段与用户问题的匹配度。

## FastAPI 工具接口

订单接口用于模拟业务系统中的动态订单数据，核心接口包括：

- `GET /health`：健康检查。
- `GET /orders/{order_id}`：按路径参数查询订单。
- `GET /orders?order_id=...`：按查询参数查询订单。
- `POST /orders/query`：Dify 订单查询分支推荐接口。
- `POST /orders/analyze`：异常处理分支推荐接口。

订单数据保存在 `order_api/orders.json` 中，字段包括订单号、客户名、当前状态、物流节点、预计送达时间、异常类型、商品类型和最后更新时间。

## Prompt 设计

项目将 Prompt 按节点拆分，降低单一 Prompt 过重带来的不稳定性：

- `prompts/rag_system_prompt.md`：约束知识库问答只能依据检索上下文回答。
- `prompts/order_query_prompt.md`：将订单接口 JSON 整理为客服可读的状态回复。
- `prompts/exception_solution_prompt.md`：综合订单事实和 SOP 输出异常处理建议。

Prompt 设计重点包括：

- 明确输入变量，避免遗漏接口结果或检索上下文。
- 固定输出结构，提高回答稳定性和可读性。
- 明确禁止编造制度、赔付金额、内部系统名称或接口中不存在的信息。
- 信息不足时直接说明，不强行补全。

## 评测设计

项目提供 `evaluation/test_questions.csv` 和 `evaluation/rag_eval.py`，用于批量调用 Dify API 并输出评测结果。

评测流程：

```text
读取测试问题 -> 调用 Dify API -> 获取回答 -> 关键词命中评分 -> 输出 CSV 结果
```

当前评测方式适合首版验证，优点是实现简单、可复现；局限是可能误判语义正确但未命中精确关键词的回答。后续可以引入语义相似度评分、LLM-as-judge 或人工抽检作为补充。

## 项目亮点

- 使用 RAG 管理物流 SOP，避免将业务规则全部硬编码在 Prompt 中。
- 使用 FastAPI 工具接口处理订单状态这类动态数据。
- 使用 Dify Chatflow 编排问题分类、参数提取、检索、HTTP 调用和回答生成。
- 使用测试集和评测脚本建立基础效果验证闭环。
- Prompt、Dify 配置和接口说明均沉淀在仓库中，便于复现和迭代。

## 可继续优化方向

- 将订单接口部署到稳定公网环境，替代临时开发隧道。
- 扩充测试集中的真实业务问法，增强分类器与检索链路的鲁棒性。
- 引入语义评测或 LLM-as-judge，补充关键词命中评测的不足。
- 增加接口鉴权、日志追踪、异常监控和权限控制，使系统更接近生产环境。
