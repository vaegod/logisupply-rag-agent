# 项目实现说明

## 1. 项目概述

本项目实现了一个面向物流供应链场景的知识库问答与 Agent 工作流系统。系统使用 Dify 构建 RAG 知识库和 Chatflow，使用 FastAPI 提供订单工具接口，并通过测试集对回答效果进行评测。

项目覆盖四项核心能力：

- 物流供应链知识库问答。
- 订单状态查询。
- 异常订单处理建议生成。
- 基于测试集的问答效果评测。

当前版本不包含前端页面、模型微调、本地大模型部署和生产级权限系统，重点放在 RAG、工具调用、工作流编排和评测闭环。

## 3. 技术栈

- Dify Knowledge：业务文档上传、切分、向量化和检索。
- Dify Chatflow：问题分类、参数提取、HTTP 工具调用和 LLM 节点编排。
- FastAPI：订单查询与异常分析工具接口。
- Python：评测脚本和接口服务。
- Markdown：业务知识库文档与 Prompt 版本管理。
- CSV：测试问题集与评测结果输出。

## 4. 业务知识库

`docs/` 目录中的 10 份物流供应链文档作为知识库来源，覆盖仓储入库、出库、冷链运输、订单延迟、货物破损、退换货、客户投诉、运输费用、库存盘点和异常工单升级规则。

文档组织原则：

- 每份文档围绕一个业务主题展开，减少多个 SOP 混写导致的检索噪声。
- 标题和段落保留业务关键词，提升检索命中率。
- 异常处理类文档尽量包含判断标准、处理步骤、升级条件和沟通要点。

## 5. Dify 工作流

Chatflow 包含四类意图：

1. 知识库问答：查询物流、仓储、冷链、退换货等业务规则。
2. 订单状态查询：查询具体订单的当前状态、节点和预计送达时间。
3. 异常处理建议：针对订单延迟、破损、温度异常等生成处理方案。
4. 其他问题：对不属于项目范围或信息不足的问题进行兜底回复。

总体流程：

```text
Start
  -> Question Classifier
  -> Knowledge Retrieval / Parameter Extractor / HTTP Request
  -> LLM
  -> Answer
```

详细节点配置见 `dify/workflow_design.md` 和 `dify/dify_config_notes.md`。

## 6. RAG 检索配置

推荐配置：

| 分支 | Retrieval Method | Top K | Score Threshold | Rerank |
|---|---|---:|---:|---|
| 知识库问答 | Hybrid Search | 5 | 0.35 | 可选 |
| 异常处理建议 | Hybrid Search | 5 | 0.35 | 推荐 |

配置思路：

- `Top K = 5` 提供较充分的召回，避免关键 SOP 片段遗漏。
- `Score Threshold = 0.35` 在召回率和噪声之间取得平衡。
- 异常处理分支需要同时结合订单事实和 SOP，开启 Rerank 有助于提升最终上下文质量。

## 7. FastAPI 订单接口

订单接口位于 `order_api/`：

- `main.py`：FastAPI 服务入口。
- `orders.json`：演示订单数据。
- `requirements.txt`：运行依赖。

核心接口：

| 接口 | 方法 | 说明 |
|---|---|---|
| `/health` | GET | 健康检查 |
| `/orders/{order_id}` | GET | 按路径参数查询订单 |
| `/orders?order_id=...` | GET | 按查询参数查询订单 |
| `/orders/query` | POST | Dify 订单查询推荐接口 |
| `/orders/analyze` | POST | 异常订单分析接口 |

本地启动：

```powershell
cd order_api
pip install -r requirements.txt
uvicorn main:app --host 0.0.0.0 --port 8000 --reload
```

Dify Cloud 需要访问公网可达的接口地址。本地开发可通过临时隧道暴露 `8000` 端口；长期演示建议使用稳定部署地址。

## 8. Prompt 版本管理

`prompts/` 目录保存 Dify LLM 节点中使用的 Prompt：

- `rag_system_prompt.md`：知识库问答。
- `order_query_prompt.md`：订单状态整理。
- `exception_solution_prompt.md`：异常处理建议。

Prompt 约束重点：

- 回答必须基于知识库上下文或接口返回。
- 动态订单信息必须来自接口结果。
- 禁止编造流程、制度、赔付金额和接口中不存在的信息。
- 信息不足时明确说明，不强行补全。
- 异常处理输出保持固定结构，便于业务人员阅读。

## 9. 评测脚本

评测相关文件：

- `evaluation/test_questions.csv`：测试问题、问题类型和期望关键词。
- `evaluation/rag_eval.py`：批量调用 Dify API 并计算关键词命中。
- `evaluation/eval_result.csv`：本地评测输出文件，不纳入版本控制。

运行方式：

```powershell
$env:DIFY_API_KEY="<DIFY_API_KEY>"
cd evaluation
python rag_eval.py
```

评测脚本会读取测试集，调用 Dify `/chat-messages` API，统计每条回答的关键词命中率，并输出 CSV 结果。

## 10. 安全与配置说明

- 仓库不保存 Dify API Key、模型密钥、临时隧道 token 或个人账号信息。
- HTTP Request 节点中的真实接口域名不写入仓库，文档中统一使用 `<order-api-domain>` 占位。
- Dify UI 中的 Prompt、节点参数和检索阈值调整后，需要同步更新 `prompts/` 或 `dify/` 文档。
- 评测输出文件 `evaluation/eval_result.csv` 已加入 `.gitignore`。

## 12. 当前边界与后续优化

当前版本适合作为 RAG + 工具调用 + 工作流编排的项目原型，主要边界包括：

- 订单数据使用 JSON 文件维护，尚未接入数据库或真实业务系统。
- 评测方式为关键词命中，尚未引入语义评分或人工复核。
- HTTP 工具接口尚未加入鉴权、监控和限流。
- Dify 工作流适合演示和验证，生产级部署仍需补充权限、日志和稳定性设计。

后续可优化方向：

- 将订单接口部署到稳定公网服务，并接入数据库。
- 扩充真实业务问法和多轮对话测试集。
- 引入语义相似度、LLM-as-judge 和人工抽检的混合评测。
- 增加接口鉴权、日志追踪、异常监控和权限控制。
