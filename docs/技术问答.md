# 技术问答

## 1. 项目解决什么问题？

项目面向物流供应链业务场景，解决三类问题：物流 SOP 和业务规则查询、具体订单状态查询、异常订单处理建议生成。系统把静态业务规则放入知识库，把动态订单状态放在工具接口中，再通过 Dify Chatflow 按问题类型选择处理路径。

## 2. 为什么使用 RAG？

物流 SOP、冷链规范、退换货规则等内容属于领域知识，直接依赖大模型已有知识容易出现幻觉。RAG 先从知识库中检索相关片段，再让模型基于片段回答，更适合企业知识库和业务规则问答场景。

## 3. 为什么还需要 FastAPI？

RAG 适合处理相对静态的文档知识，但订单状态是动态数据。FastAPI 订单接口用于提供订单当前状态、物流节点、预计送达时间和异常类型，让 Dify 通过 HTTP Request 获取实时业务数据。

## 4. 这个项目为什么可以称为 Agent 工作流？

项目不属于完全自主规划型 Agent，而是工作流型 Agent 应用。它具备意图识别、参数提取、知识检索、工具调用和基于工具结果生成回答等能力，执行路径由 Dify Chatflow 预先编排。

## 5. 三条核心业务分支是什么？

- 知识库问答：问题分类后进入 Knowledge Retrieval，再由 LLM 基于上下文回答。
- 订单状态查询：提取订单号后调用 `/orders/query`，再由 LLM 整理订单状态。
- 异常处理建议：提取订单号后调用 `/orders/analyze`，再检索异常 SOP，最后由 LLM 生成处理建议。

## 6. Question Classifier 的作用是什么？

Question Classifier 用于判断用户问题属于知识库问答、订单状态查询、异常处理建议或其他问题。分类结果会决定后续节点路径，因此分类描述和示例需要覆盖常见业务表达。

## 7. Parameter Extractor 的作用是什么？

Parameter Extractor 用于从自然语言中提取结构化参数，例如从“订单 JD2026001 到哪了”中提取：

```json
{
  "order_id": "JD2026001"
}
```

后续 HTTP Request 节点会使用该参数调用订单接口。

## 8. 为什么订单查询推荐使用 POST？

Dify 的 GET URL 路径或查询参数在不同版本中可能出现变量拼接兼容问题。使用 `POST /orders/query` 并通过 JSON Body 传递 `order_id`，变量插入更稳定，也便于扩展更多查询参数。

## 9. 异常处理为什么同时使用接口和知识库？

异常处理需要两类信息：订单接口提供当前订单事实，例如异常类型和物流节点；知识库提供处理规则，例如延迟、破损、温控异常对应的 SOP。两类信息结合后，模型才能生成可执行的处理建议。

## 10. 项目如何减少幻觉？

- 业务规则通过 RAG 检索，不直接依赖模型记忆。
- 动态订单状态通过接口获取，不写死在 Prompt 或知识库中。
- Prompt 明确禁止编造制度、金额、流程和接口中不存在的信息。
- 信息不足时要求模型说明未找到依据。
- 使用测试集和 badcase 分析检查回答是否偏离事实来源。

## 11. Top K、Score Threshold 和 Rerank 如何设置？

项目推荐知识问答和异常处理分支使用 `Top K = 5`、`Score Threshold = 0.35`。异常处理分支可开启 `Qwen/Qwen3-Reranker-8B` 重排，提升召回片段与问题的相关性。

## 12. Prompt 如何拆分？

项目将 Prompt 拆为三类：

- 知识库问答 Prompt：约束模型仅依据检索上下文回答。
- 订单查询 Prompt：将接口 JSON 整理为中文状态回复。
- 异常处理 Prompt：结合接口结果和 SOP 输出处理建议。

这种拆分方式让每个节点职责更清晰，也便于单独调试。

## 13. 评测如何实现？

`evaluation/test_questions.csv` 保存测试问题、问题类型和期望关键词。`evaluation/rag_eval.py` 批量调用 Dify `/chat-messages` API，获取回答后计算关键词命中率，并将结果写入 `evaluation/eval_result.csv`。

## 14. 关键词评测有什么局限？

关键词评测实现简单、便于自动化，但对表达变化比较敏感。部分答案语义正确但没有命中精确关键词时，可能被误判为失败。后续可加入语义相似度、LLM-as-judge 或人工抽检。

## 15. Dify 联调中需要注意什么？

- Dify Cloud 无法直接访问本机 `127.0.0.1`，订单接口需要使用公网可达地址。
- HTTP URL、POST Body 和 Prompt 中优先使用变量插入器。
- 修改 Chatflow 后需要重新发布，评测脚本调用的是已发布版本。
- 不要将 Dify API Key、模型密钥或临时隧道地址提交到仓库。

## 16. 项目和普通聊天 Demo 的区别是什么？

项目不仅调用模型生成回答，还包含知识库构建、问题分类、参数提取、HTTP 工具调用、Prompt 约束、接口联调和自动化评测，能体现一个大模型应用从业务输入到效果验证的闭环。

## 17. 当前版本的边界是什么？

当前版本重点验证 RAG + 工具调用 + 工作流编排链路，不包含前端页面、模型微调、本地大模型部署和生产级权限系统。订单数据使用 JSON 文件维护，适合演示和原型验证。

## 18. 后续如何工程化？

后续可将订单数据接入数据库或真实业务系统，增加接口鉴权、请求日志、错误监控、权限隔离和稳定部署；评测侧可增加语义评分、多轮对话评测和人工复核样本。
